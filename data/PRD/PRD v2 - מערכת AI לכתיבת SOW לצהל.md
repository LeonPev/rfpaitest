## הבעיה

תהליך כתיבת SOW ברכש צה"ל אורך **12 חודשים** מקצה לקצה. הסיבות:

- **מחסור בספקים** — תנאים מגבילים מדי שמונעים השתתפות בשוק
- **מפרטים לא ברורים** — פרשנויות שונות בין הגורמים; ערעורים בסיבובי אישור
- **מחירים ותנאים לא ריאליים** — אין reference לנורמות שוק בזמן כתיבה
- **חוסר כיסוי סיכונים** — חשיפה משפטית, תפעולית, ביטחונית שמתגלה מאוחר

כיום אין שום כלי שעוזר למנהל הרכש לזהות את הכשלים האלה לפני שה-SOW יוצא לדרך.

## חזון המוצר

> מערכת AI שמקבלת נקודת התחלה — ומחזירה SOW מוכן, עם כל ההסברים, ההשוואות, והנימוקים — כך שמנהל הרכש מבין בדיוק מה לקבל, מה לדחות, ולמה.

המטרה אינה להחליף את מנהל הרכש. המטרה להפוך אותו לגרסה הכי טובה של עצמו.

## ה-Workflow — מה המערכת עושה

### טריגר
המשתמש פותח תהליך חדש כי:
- חוזה קיים מסתיים ודורש חידוש
- צורך רכש חדש עלה

### שלב 1 — קלט
המשתמש מספק נקודת התחלה:
- SOW קודם (אופציונלי)
- מסמכים נוספים — דרישות פנימיות, מפרטים, ספקים מוכרים (אופציונלי)
- דגשים חופשיים בטקסט (אופציונלי)
- אפשר גם: לא לתת כלום — המערכת מתחילה מקטגוריה בלבד

### שלב 2 — אורקסטרציה מרובת סוכנים

```
                    [אורקסטרטור]
                         |
          +--------------+--------------+
          |              |              |
   [הבנת הקשר]    [סקר שוק]      [סקר ספקים]
   קריאת קלט,    חיפוש ממשלות   מציאת ספקים
   הגדרת צרכים   אחרות + מחירים  בארץ ובחו"ל
          |              |              |
          +--------------+--------------+
                         |
               [אינטגרציה + כתיבה]
               שימוש בעבר + חומר חיצוני
               → דראפט ראשוני
                         |
          +--------------+--------------+
          |              |              |
       [חוק]        [סיכונים]      [עסק]
    רגולציה,        חשיפה,         היגיון
    ציות            ביטחון         מסחרי
          |              |              |
          +--------------+--------------+
                         |
               [שופט ומדרג]
               מה חייב לשנות / מומלץ / אופציונלי
                         |
               [אורקסטרטור מרכז]
               בוחן שה-output מוכן למשתמש
                         |
                  [פלט למשתמש]
```

**עקרון מפתח:** המשתמש מביא **א** — מקבל **ב**. ב-ב מוסבר כל שינוי מול א, כולל למה.

## חוויית משתמש — שתי אפשרויות ל-MVP

*(ההחלטה תלויה — ראה החלטות פתוחות)*

### אפשרות א — Wizard
תהליך רב-שלבי ומובנה. המשתמש עובר שלב אחרי שלב עם שאלות, בחירות, ותפריטים.

| יתרון | חיסרון |
|---|---|
| ברור ומונחה גם למשתמש לא מנוסה | פחות גמישות; צריך לדעת מראש מה לשאול |
| שולט בפורמט הקלט → output איכותי יותר | זמן כניסה ארוך יותר |
| קל להנחות כיסוי של כל הנושאים | מגביל מי שיודע מה הוא רוצה |

### אפשרות ב — Chat
ממשק שיחה חופשי. המשתמש כותב בשפה טבעית, שואל שאלות, מדייק.

| יתרון | חיסרון |
|---|---|
| גמיש ומהיר למי שיודע מה הוא רוצה | עלול לבלבל מי שלא יודע מאיפה להתחיל |
| תחושה טבעית; מאפשר דו-שיח | הקלט פחות מובנה → output יכול להיות פחות שלם |
| קל יותר לדייק בדרישות | פחות "guidance" מובנה |

## הפלט — מה המשתמש מקבל

המסך מציג **השוואה:** מה הבאת ← מה אנחנו מציעים.

כל שינוי מוצג עם:
- **מה השתנה** — הטקסט המקורי לעומת המוצע
- **למה** — מקור השינוי: סקר שוק / חוק ורגולציה / ניתוח ספקים / פרקטיקה ממשלה אחרת
- **רמת חשיבות** — 🔴 חייב / 🟡 מומלץ / 🟢 אופציונלי
- **מקורות** — קישורים לאתרים, מסמכים, ממשלות שהיוו בסיס להמלצה

המשתמש יכול:
- **לאשר** שינוי בלחיצה
- **לדחות** שינוי עם הסבר (אופציונלי)
- **להתעמק** — לפתוח פירוט מורחב, לשאול AI שאלות על ההמלצה, לצלול למקורות

לאחר סיום — ייצוא SOW מסוכם, עם changelog של מה התקבל ומה לא.

## למידה ושיפור מתמשך

**מה נשמר מכל פעילות משתמש:**
- כל שינוי שהתקבל / נדחה (ומה ההסבר אם ניתן)
- זמן עבודה, מה נצפה, מה נוצר
- בקשות חוזרות שלא קיבלו מענה ("מה המשתמש ביקש ולא קיבל")
- נטישות — סשנים שהתחילו ולא הושלמו, יחידות שלא חוזרות

**סוכן למידה:** רץ על כל הלוגים האלה, מזהה דפוסים חוזרים, מתעלם מאנומליות, ומזין עדכון לזיכרון המערכת.

**כלל:** לא כל פידבק הוא אמת — הסוכן מחפש **חזרתיות** לפני שמשנה כיוון.

## מדדי הצלחה

| מה מודדים | יעד |
|---|---|
| זמן ממוצע לכתיבת SOW (ראשוני) | מ-שבועות → יום עבודה |
| % SOWs שעוברים אישורים בסיבוב ראשון | >70% |
| % שינויים מוצעים שהתקבלו | מדד כיוון — עולה עם הזמן |
| משתמשים פעילים שבועיים | עולה עקבית לאחר השקה |
| אינגייג'מנט ממוצע לסשן | >15 דקות (מראה שהמוצר שימושי) |

> **חסר קריטי:** נתוני outcome — לאיזה SOWs הגיעו ספקים, אילו עברו אישורים. **חייב מנגנון לאיסוף זה מהיום הראשון.**
