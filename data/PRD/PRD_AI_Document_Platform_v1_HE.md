# PRD: פלטפורמת מסמכים חכמה מבוססת בינה מלאכותית

**גרסה:** 1.0
**תאריך:** 10 במרץ 2026
**סטטוס:** טיוטה

---

## 1. תקציר מנהלים

פלטפורמת AI שמקבלת קלטים מרובי-קבצים מורכבים (SOW, ניתוח שוק, שאלות ותשובות, מכרזים ועוד), מבינה לעומק את כוונת המשתמש באמצעות שאלות איטרטיביות, מתכננת את העבודה עם אישור המשתמש, ואז מבצעת באמצעות סוכנים מתמחים כדי לייצר פלטי מסמכים מוערים וניתנים לסקירה בפורמטים מרובים.

**הצעת ערך מרכזית:** הפיכת שבועות של עבודת מסמכים ידנית לשעות של יצירה מונחית בסיוע AI, עם מעקב מלא ושליטה אנושית בכל שלב.

---

## 2. הגדרת הבעיה

צוותי רכש ממשלתיים וביטחוניים משקיעים זמן רב ב:
- עדכון ידני של מסמכי SOW בין תקופות חוזה
- הצלבת דרישות רגולטוריות, משפטיות ותחומיות
- יצירת מסמכים נלווים (ניתוח שוק, שאלות ותשובות, מיפוי ספקים)
- סקירה ומעקב שינויים ללא נימוק ברור
- המרת מסמכים סופיים בין פורמטים מרובים

הכלים הקיימים חסרים מודעות תחומית, עקיבות שינויים, ויכולת לתזמר זרימות עבודה רב-מסמכיות עם סקירה אנושית בלולאה.

## 4. זרימת עבודה מרכזית

```
[1] קלט           [2] הבנה            [3] תכנון          [4] ביצוע          [5] סקירה          [6] ייצוא
 המשתמש מעלה  ->  AI שואל שאלות  ->  AI מציג תוכנית ->  סוכנים מרובים ->  סקירה ברמת    ->  PDF, DOCX,
 קבצים מרובים     עד >99%            עבודה עם           ביצוע מקבילי      סעיף עם           MD, HTML,
 + הקשר           ביטחון              תוצרים                                הערות              GDOC, הדפסה
```

### פירוט שלבים

**שלב 1 - קלט:** המשתמש מעלה קובץ אחד או יותר (כל שילוב של SOW, ניתוח שוק, שאלות ותשובות, הצעות, תקנות וכו') ומספק תיאור טקסטואלי של מה שהוא צריך.

**שלב 2 - הבנה:** ה-AI מנתח את כל הקבצים, מזהה פערים ועמימויות, ואז שואל שאלות ממוקדות בסבבים עד שמגיע ל->99% ביטחון שהוא מבין את הבקשה. כל סבב צריך להפחית עמימות ב-50% לפחות.

**שלב 3 - תכנון:** ה-AI מציג תוכנית עבודה מובנית: אילו קבצים ייווצרו/יעודכנו, אילו סוכנים יטפלו במה, מספר סעיפים משוער למסמך, וכל הנחה שנעשתה. המשתמש מאשר, מתקן או דוחה.

**שלב 4 - ביצוע:** תוכנית מאושרת מפעילה סוכנים מתמחים (תחום, משפט, איכות, מחקר שוק וכו') שעובדים במקביל היכן שאפשר. סטרימינג התקדמות בזמן אמת למשתמש.

**שלב 5 - סקירה:** כל סעיף שנוצר/עודכן מציג:
- את התוכן עצמו
- הערת צד שמסבירה *למה* זה נכתב/שונה/נוסף/נמחק
- כפתורי פעולה: אישור, מחיקה, עידון, פעולת AI
- יכולת בחירה מרובה לפעולות אצווה

**שלב 6 - ייצוא:** לאחר שהמשתמש מאשר שינויים, ייצוא ל: PDF, Markdown, Google Docs, DOCX, HTML, פורמט מוכן להדפסה.

## 5. סיפורי משתמש (User Stories)

### P0 - חובה (חוסמי השקה)

| **P0-02** | כמשתמש, אני יכול לתאר מה אני צריך בטקסט חופשי לצד הקבצים שהעליתי | שדה קלט טקסט ללא הגבלת תווים. המערכת שומרת הקשר לצד קבצים לשימוש ה-AI. |
| **P0-03** | כמשתמש, ה-AI שואל אותי שאלות הבהרה לפני תחילת העבודה, וממשיך לשאול עד שהוא מבין את בקשתי במלואה | AI מייצר שאלות ממוקדות בכל סבב. כל סבב מצמצם היקף. מינימום סבב אחד, ממשיך עד ש-AI מדווח על >99% ביטחון. המשתמש יכול לומר "פשוט תתחיל" כדי לדלג. |
| **P0-04** | כמשתמש, אני יכול לראות את תוכנית העבודה של ה-AI לפני שהוא מבצע, כולל רשימת קבצים שייווצרו/יעודכנו ומה כל סוכן יעשה | תוכנית מוצגת כרשימה מובנית: שם קובץ, מטרה, סעיפים לייצור, סוכן מוקצה. המשתמש יכול לאשר, לערוך או לדחות את התוכנית. |
| **P0-05** | כמשתמש, אני יכול לראות התקדמות בזמן אמת כאשר הסוכנים מבצעים את התוכנית | סטרימינג התקדמות: איזה סוכן פעיל, איזה סעיף נוצר, % השלמה למסמך. |
| **P0-06** | כמשתמש, אני יכול לסקור כל סעיף בקובץ שנוצר/עודכן עם הערה המסבירה את הנימוק לשינוי | לכל סעיף פאנל צד שמציג: סוג שינוי (נוסף/שונה/נמחק/ללא שינוי), טקסט נימוק, הפניה למקור (איזה קובץ קלט או הוראת משתמש הניע את זה). |
| **P0-07** | כמשתמש, אני יכול לאשר, למחוק או לעדן כל סעיף בנפרד | אישור: נועל סעיף כסופי. מחיקה: מסיר סעיף. עידון: פותח עורך אינליין לעריכות ידניות. סטטוס נשמר בין סשנים. |
| **P0-08** | כמשתמש, אני יכול לבקש מה-AI לבצע פעולה על סעיף ספציפי (לדוג': "קצר את זה", "הוסף הפניה רגולטורית", "תרגם") | קלט פעולת AI בטקסט חופשי לכל סעיף. AI מייצר מחדש את הסעיף בלבד, משמר את השאר. גרסה חדשה מחליפה ישנה עם הערה מעודכנת. |

## 6. מערכת ביטחון ושאלות (פירוט)

שלב השאלות של ה-AI הוא מבדל מרכזי. כך זה צריך לעבוד:

### מודל ביטחון

```
סבב 1: ניתוח קבצים + שאלות ראשוניות      -> יעד: 60-70% ביטחון
סבב 2: המשך בהתבסס על תשובות              -> יעד: 80-90% ביטחון
סבב 3: מקרי קצה ואישור                     -> יעד: 95-99% ביטחון
סבב N: רק אם נותרה עמימות קריטית           -> יעד: >99% ביטחון
```

### קטגוריות שאלות

| קטגוריה | דוגמה | מתי לשאול |
|----------|-------|----------|
| **היקף** | "העלית SOW וניתוח שוק. האם ה-SOW הפלט צריך לשלב ממצאים מניתוח השוק, או שאלו תוצרים עצמאיים?" | תמיד (סבב 1) |
| **כוונה** | "ה-SOW מזכיר טעינת רכב חשמלי כחדש. האם להרחיב למפרט טכני מלא, או לשמור ברמת המסמך הקיים?" | כשלקלט יש סעיפים חדשים/עמומים |
| **אילוצים** | "האם יש תקנים רגולטוריים ספציפיים שחובה לציין? מדיניות פנימית שצריך לעקוב אחריה?" | כשמדובר בעמידה תחומית ספציפית |
| **פלט** | "אתה צריך 3 קבצי פלט. האם הם צריכים להצליב זה את זה, או להיות מסמכים עצמאיים?" | כשמבוקשים פלטים מרובים |
| **עדיפות** | "מספר סעיפים ניתנים לשיפור. אילו תחומים חשובים ביותר: עמידה משפטית, דיוק טכני, או קריאות?" | כשקיימים שיפורים פוטנציאליים רבים |
| **פורמט** | "האם הפלט צריך לעקוב אחר המבנה המדויק של מסמך הקלט, או שאפשר לארגן מחדש לבהירות?" | כשמבנה הקלט אינו אופטימלי |

### מנגנון יציאה
המשתמש תמיד יכול לומר "פשוט תתחיל עם מה שאתה יודע" כדי לדלג על שאלות נותרות. במקרה זה, ה-AI מתעד את ההנחות שלו בתוכנית העבודה לסקירת המשתמש.

## 7. ארכיטקטורת סוכנים (פירוט)

### סוגי סוכנים

| סוכן | אחריות | רץ מתי |
|-------|---------|--------|
| **מתזמר (Orchestrator)** | מפרק תוכנית למשימות, מקצה לסוכנים, מרכיב פלט סופי | תמיד |
| **מומחה תחום** | מייצר/מעדכן תוכן עם ידע תחומי (דלק, רפואי, מזון וכו') | תמיד |
| **סקירה משפטית** | מסמן סיכונים, בודק עמידה רגולטורית, מאמת שפה משפטית | כשלמסמך יש תוכן חוזי/משפטי |
| **אבטחת איכות** | בדיקת עקביות, אימות הפניות לתקנים, בדיקת שלמות | תמיד |
| **מחקר שוק** | מזהה ספקים, מעריך היתכנות, ממפה נוף תחרותי | כשמתבקש ניתוח שוק |
| **כותב הערות** | מייצר הערות נימוק לכל שינוי סעיף | תמיד (לאחר יצירה) |
| **כותב סיכום** | מפיק סיכום עבודה, דוח ביטחון והמלצות | תמיד (לאחר יצירה) |

### מודל ביצוע
- סוכנים רצים במקביל כשהם עצמאיים (לדוג': מומחה תחום + מחקר שוק)
- סדרתי כשתלויים (לדוג': מומחה תחום -> סקירה משפטית -> אבטחת איכות)
- כל הסוכנים מזינים תוצאות למתזמר להרכבה
- אירועי התקדמות בזמן אמת נשלחים לממשק המשתמש באמצעות SSE

## 10. שיקולים טכניים

### תשתית קיימת (מה-POC הנוכחי)
- **צד שרת:** Python Flask עם SSE סטרימינג
- **AI:** Google Gemini 2.5-Flash (צינור רב-סוכנים)
- **צד לקוח:** Vanilla JS עם תמיכת RTL לעברית
- **אחסון:** שמירת פרויקטים מבוססת JSON
- **סוכנים:** 6 סוכנים מתמחים כבר מיושמים (תחום, משפטי, איכות, מתזמר, צ'אט, שוק)

### החלטות טכניות נדרשות

| החלטה | אפשרויות | המלצה |
|-------|----------|-------|
| ספק AI | Gemini (נוכחי), Claude API, היברידי | הערכת Claude לאיכות הערות; Gemini למהירות. היברידי עשוי להיות אופטימלי. |
| מנוע ייצוא | python-docx + מותאם, Pandoc, שירות ייעודי | Pandoc לרב-פורמטים; python-docx להתאמת DOCX. |
| אינטגרציית Google Docs | Google Docs API, ייצוא-והעלאה, סנכרון חי | Google Docs API ל-P2. ייצוא-והעלאה כפתרון ביניים. |

| ניקוד ביטחון | היוריסטיקה מבוססת כללים, הערכה עצמית LLM, היברידי | התחלה עם הערכה עצמית LLM (P0), הוספת אימות היוריסטי (P1). |
