## 1. סיכום מנהלים

### הבעיה
תהליך כתיבת מסמכי SOW (Statement of Work) ברכש צה"ל אורך כיום **12 חודשים** מקצה לקצה, עם מחזור חידוש חוזי של **חודש לפחות**. התהליך סובל מכשלים חוזרים: מחסור בספקים, מפרטים טכניים לא ברורים, תנאים ומחירים לא ריאליים, וחוסר בכיסוי סיכונים מספיק. כשלים אלה גוררים עיכובים, ערעורים, וחשיפה משפטית ותפעולית.

### הפתרון
מערכת AI המיועדת לאנשי רכש צה"ל, שיודעת **לנסח SOW מלא**, **לנתח ולציין** אותו, **לזהות סיכונים**, **לחקור את השוק**, **למצוא ספקים**, **להשוות מול SOWs ממשלות אחרות**, ולהציג למשתמש **תכנית פעולה ממוקדת** להגיע ל-SOW ברמת הצלחה מרבית — הכל ממשק אחד, בדפדפן.

### המוצר
כלי Web-based, ענן, נגיש לכל מנהל רכש, ללא תלות במערכות פנים-ארגוניות. מתחיל כמוצר אזרחי עצמאי ומתפתח בהתאם לדרישות צה"ל.

### לוח זמנים
| אבן דרך | מועד |
|---|---|
| POC — הוכחת היתכנות | סוף שבוע זה |
| MVP — קטגוריית דלק | תוך חודש (אפריל 2026) |
| השקה רשמית | תחילת קיץ 2026 |

## 2. הגדרת הבעיה

### ההקשר הנוכחי
רכש צה"ל מפעיל עשרות מחזורי רכש במקביל, בעשרות קטגוריות (מזון, ציוד רפואי, רכב ודלק, ריהוט, חומרי ניקוי ועוד). כל מחזור מתחיל בכתיבת SOW — מסמך הדרישות שהספק נדרש לעמוד בהן.

כיום, **כתיבת SOW היא תהליך ידני לחלוטין**, תלוי בניסיון האישי של מנהל הרכש, בגישה למסמכים קודמים, ובהדרכה פנימית לא אחידה.

### כשלים מרכזיים שזוהו

| # | כשל | השפעה |
|---|---|---|
| 1 | **מחסור בספקים** — הגדרות צרות מדי מונעות השתתפות | מכרז יוצא לדרך ואין מגיבים; עיכוב של חודשים |
| 2 | **מפרט טכני לא ברור** — אפשרות לפרשנויות סותרות | ערעורים, מחלוקות, שינויי חוזה יקרים |
| 3 | **מחירים ותנאים לא ריאליים** | ספקים לא מגישים הצעה, או מגישים ומפסידים כסף → ביצוע ירוד |
| 4 | **חוסר כיסוי סיכונים** | חשיפה משפטית, תפעולית, ביטחונית |
| 5 | **מחזור אישור ארוך** | 3 רמות אישור (מדור → רכש/משביר → משרד הביטחון) — כל אחת עם מספר בוחנים |
| 6 | **היעדר למידה בין-מחזורית** | אין מנגנון ממסד לשמר ידע שנצבר; כל מחזור מתחיל כמעט מאפס |

### נקודות כאב של המשתמשים
- "לא יודעים מה חסר עד שמישהו מעיר בסיבוב אישור"
- "לוקח שבועות להתאים SOW ישן לצרכים חדשים"
- "אין לנו כלי להשוות מחירי שוק לפני שנורמר תנאי"
- "אין לנו benchmark — מה SOW טוב בקטגוריה הזו?"

## 3. חזון המוצר

> **"כל מנהל רכש, לא משנה ניסיונו, יוכל להפיק SOW ברמת מקצוענות עולמית — תוך שעות, לא חודשים."**

המערכת תשמש כ**יועץ רכש AI בכיר** שיושב לצד מנהל הרכש בכל שלב:
1. **מנסח** — כותב טיוטה ראשונה על בסיס קטגוריה ופרמטרים
2. **מנתח** — מצביע בדיוק על מה חסר, לא ברור, או מסוכן
3. **מחקר** — מביא נתוני שוק אמיתיים ומסמכי ייחוס ממשלתיים
4. **מדרג ספקים** — מציג שמות ספקים רלוונטיים לפני פרסום
5. **לומד** — משתפר כל פעם מפידבק מפורש ומהתנהגות משתמשים

## 4. קהל יעד ופרסונות

### פרסונה ראשית — מנהל/ת רכש
| פרמטר | פירוט |
|---|---|
| **תפקיד** | קב"ת / מנהל רכש ברמת מדור |
| **ניסיון טכנולוגי** | ממוצע-נמוך; עובד עם Word, Excel, מייל |
| **מטרה עיקרית** | לצאת עם SOW שיאושר מהר ויביא ספקים טובים |
| **כאב עיקרי** | חוסר ידיעה "מה חסר לי" ופחד מהסיבוב השני-שלישי של הערות |
| **הצלחה** | SOW עובר אישורים בסיבוב ראשון, ספקים מגישים הצעות |

### פרסונה משנית — גורם מאשר (מדור / משביר / משרד הביטחון)
| פרמטר | פירוט |
|---|---|
| **תפקיד** | בוחן SOW ברמת מדור, רכש, או משרד הביטחון |
| **מטרה עיקרית** | לוודא שה-SOW שלם, תקין ולא מחשיף לסיכון |
| **הצלחה** | מסמכים מגיעים מוכנים, פחות הערות חוזרות |

> **הערה לצוות:** בשלב ראשון (MVP) נתמקד בפרסונה הראשית בלבד.

### יעדי מוצר

| # | יעד | מדד הצלחה | יעד כמותי |
|---|---|---|---|

| O4 | כיסוי סיכונים | ציון כיסוי-סיכון ממוצע ל-SOW | >85/100 |
| O5 | אימוץ מוצר | MAU (משתמשים פעילים חודשיים) | 80% ממנהלי הרכש הרלוונטיים |

### מדדי AI / איכות

| מדד | הגדרה |
|---|---|
| **SOW Completeness Score** | ציון 0–100 לכל SOW על בסיס checklist דינמי |
| **Risk Coverage Rate** | % סעיפי סיכון שכוסו מתוך רשימה מוגדרת |
| **Market Alignment Score** | מידת ההלימה בין תנאי SOW לנורמות שוק |
| **Model Feedback Rate** | % תגובות "אגודל למטה" מתוך כלל האינטראקציות |
| **User Retention** | % משתמשים שחוזרים שבוע אחרי שבוע |

## 6. פתרון — סקירה כללית

### עקרונות עיצוב

1. **Zero-installation** — גישה דרך דפדפן בלבד, ללא התקנה
2. **Human-in-the-loop** — ה-AI מציע, המשתמש מחליט
3. **Explainability-first** — כל המלצה מגיעה עם הסבר ומקור
4. **Iterative improvement** — כל שימוש מלמד את המערכת
5. **Minimal dependencies** — לא דורש התממשקות עם מערכות IDF בשלב ראשון

### מחזור חיים של SOW במערכת

```
[כניסה] → [בחר קטגוריה] → [הגדר פרמטרים] → [AI מייצר טיוטה]
       ↓
[ניתוח + ציון] → [המלצות שיפור] → [מחקר שוק + ספקים]
       ↓
[השוואה ל-SOWs ממשלות אחרות] → [SOW מעודכן]
       ↓
[תוכנית פעולה לאישורים] → [ייצוא + שמירה] → [פידבק]
       ↓
              [לולאת למידה]
```

### מודול 1 — יצירת SOW (SOW Generator)

#### תיאור
המשתמש בוחר קטגוריה ומזין פרמטרים בסיסיים. המערכת מייצרת טיוטת SOW מלאה בעברית, בפורמט תקני של רכש צה"ל.

#### דרישות פונקציונליות

| ID | דרישה | עדיפות |
|---|---|---|
| SOW-01 | משתמש בוחר קטגוריה מרשימה (ולאחר מכן גם מחיפוש חופשי) | P0 |
| SOW-02 | שאלון קצר (Wizard) שואל על: כמות משוערת, גיאוגרפיה, תדירות, מפרט מיוחד, דרישות ביטחון | P0 |
| SOW-03 | המערכת מייצרת טיוטה מלאה בעברית עם כל הפרקים הסטנדרטיים | P0 |
| SOW-04 | הפרקים כוללים לפחות: מבוא, הגדרות, מפרט טכני, תנאי אספקה, איכות, חירום, ביטחון, פיקוח ובקרה, קטלוג | P0 |
| SOW-05 | המשתמש יכול לערוך כל פסקה ישירות בממשק | P0 |
| SOW-06 | ניתן לטעון SOW קיים (Word/PDF) לצורך ניתוח ושיפור | P1 |
| SOW-07 | ניתן לבחור "בסס על SOW קודם" מרשימת SOWs שנשמרו | P1 |
| SOW-08 | תבנית הפלט ניתנת לייצוא כ-Word (.docx) ו-PDF | P0 |

#### לוגיקת ה-AI
- מודל שפה (LLM) מבוסס על: (א) SOWs קיימים שהוזנו למערכת כ-few-shot examples, (ב) prompt engineering קטגורי-ספציפי
- כל קטגוריה תקבל context window מותאם עם SOWs לדוגמה

### מודול 2 — ניתוח וציון SOW (SOW Analyzer & Scorer)

#### תיאור
לב המערכת. מנתח כל SOW ומחזיר ציון כולל + פירוט per-section + רשימת פעולות מומלצות.

#### דרישות פונקציונליות

| ID | דרישה | עדיפות |
|---|---|---|
| ANA-01 | ניתוח אוטומטי של כל SOW שנוצר או הועלה | P0 |
| ANA-02 | ציון כולל מ-0 עד 100 ("SOW Completeness Score") | P0 |
| ANA-03 | פירוט ציון לכל פרק בנפרד | P0 |
| ANA-04 | רשימת "פעולות נדרשות" ממוינת לפי קריטיות (Critical / Warning / Suggestion) | P0 |
| ANA-05 | כל הערה מגיעה עם: הסבר, מה הסיכון אם לא מטופל, דוגמה לניסוח טוב | P0 |
| ANA-06 | ניתוח שפה — זיהוי מונחים עמומים ("בהתאם לנדרש", "ככל שניתן") ודגלום | P1 |
| ANA-07 | ניתוח עקביות — זיהוי סתירות בין פרקים שונים | P1 |
| ANA-08 | השוואת SOW לנורמות קטגוריה (benchmark) — מה ה-SOW הממוצע בקטגוריה נותן | P1 |
| ANA-09 | ציון משתנה ב-real-time ככל שהמשתמש עורך | P2 |

#### מבנה הציון (Scoring Rubric — גרסה ראשונית)

| קטגוריית ציון | משקל | פרמטרים |
|---|---|---|
| שלמות מבנית | 25% | כל הפרקים הנדרשים קיימים |
| בהירות מפרט טכני | 20% | ספציפיות, מדידות כמותיות |
| ריאליות תנאים | 20% | זמני אספקה, מינימום הזמנה, קנסות |
| כיסוי סיכונים | 20% | חירום, ביטחון, עיצומים, אחריות |
| ידידותיות לספקים | 15% | בהירות תנאי השתתפות, פתיחות שוק |

> **הערה:** ה-Rubric הוא דינמי ומתעדכן עם למידה. בשלב MVP — מוגדר ידנית על בסיס ניסיון מומחים וקריאת SOWs קיימים.

### מודול 3 — זיהוי וניהול סיכונים (Risk Engine)

#### תיאור
מנוע ייעודי לזיהוי סיכונים בSOW על פי קטגוריות סיכון מוגדרות.

#### דרישות פונקציונליות

| ID | דרישה | עדיפות |
|---|---|---|
| RISK-01 | זיהוי אוטומטי של סיכונים מ-taxonomy מוגדר (ראה להלן) | P0 |
| RISK-02 | כל סיכון מוצג עם: רמת חומרה, הסבר, ניסוח מומלץ לסעיף | P0 |
| RISK-03 | "Risk Coverage Dashboard" — ויזואליזציה של כמה מתוך סיכוני הקטגוריה כוסו | P0 |
| RISK-04 | ספציפי לקטגוריה — סיכוני דלק שונים מסיכוני ציוד רפואי | P0 |
| RISK-05 | עדכון taxonomy הסיכונים דרך ממשק Admin | P1 |

#### Taxonomy סיכונים — ראשוני

| קטגוריה | סיכונים לכיסוי |
|---|---|
| **אספקה** | מחסור ספק, אי-עמידה בלוחות זמנים, תלות בספק יחיד |
| **מחיר** | תנודתיות מחיר, חוסר הצמדה, מניפולציה |
| **איכות** | חוסר מנגנון בדיקה, אין סנקציות על פגם |
| **ביטחוני** | גישה לא מורשית, דרישות אזרחות/סיווג לא מוגדרות |
| **משפטי** | הגדרת "כוח עליון", סמכות שיפוט, אחריות מוגבלת |
| **חירום** | אין נוהל חירום, אין מלאי חירום, אין SLA חירום |
| **סביבתי/בטיחות** | (ספציפי לדלק: MSDS, אחסון, פינוי) |

#### דרישות פונקציונליות

| ID | דרישה | עדיפות |
|---|---|---|
| MKT-01 | חיפוש אוטומטי של מחירי שוק לקטגוריה הנוכחית (Google, מאגרים מסחריים) | P0 |
| MKT-02 | רשימת ספקים פוטנציאליים לקטגוריה — שם, אתר, תיאור קצר | P0 |
| MKT-03 | כל ספק מוצג עם: מקור המידע, תאריך מציאה, רמת אמינות | P0 |
| MKT-04 | ניתוח "ריאליות תנאים" — השוואה בין תנאי SOW לנורמות שוק שנמצאו | P1 |
| MKT-05 | "Market Gap Report" — תנאים ב-SOW שסוטים משמעותית מהשוק (גבוה/נמוך) | P1 |
| MKT-06 | רשימת ספקים ניתנת לעריכה ידנית (הוספה/הסרה) על ידי משתמש | P1 |
| MKT-07 | שמירת ספקים לפי קטגוריה בבסיס ידע פנימי ומתפתח | P1 |

#### מקורות נתונים

| סוג | מקורות |
|---|---|

| **פלטפורמות B2B** | Alibaba, Thomasnet, GlobalSpec (לציוד) |
| **אתרי מכרזים** | מרכב"ה, מינהל הרכש הממשלתי, רכש.gov.il |

## 9. ארכיטקטורה טכנית

### עקרונות

1. **Cloud-first** — שלב MVP פרוס על ענן מסחרי (AWS / GCP / Azure)
2. **מינימום dependencies** — לא מתממשק עם מערכות IDF בשלב ראשון
3. **Stateless API** — Backend כ-REST / GraphQL API
4. **Modular** — כל מנוע (ניתוח, שוק, ספקים) הוא service עצמאי

### Stack מוצע

| שכבה | טכנולוגיה | נימוק |
|---|---|---|
| **Frontend** | React / Next.js | RTL support, SSR, מהיר לפיתוח |
| **Backend** | Python (FastAPI) | אקוסיסטם AI עשיר |
| **AI / LLM** | Claude API (Anthropic) | ביצועים בעברית, context window גדול |
| **Web Scraping** | Playwright + Apify | גמישות בסריקת אתרים |
| **Search** | Google Custom Search API | מחקר שוק ו-reference SOWs |
| **DB** | PostgreSQL + pgvector | שמירת SOWs + Vector search |
| **Storage** | S3 / GCS | קבצים (Word, PDF) |
| **Auth** | Auth0 / Supabase Auth | מהיר ל-MVP |
| **Infra** | AWS / Docker / Vercel | גמישות |

### שכבת AI

```
[User Input]
    ↓
[Orchestration Layer — LangChain / LlamaIndex]
    ↓
[Claude API] ← [SOW Templates Context] ← [Vector DB של SOWs קיימים]
    ↓
[Post-processing: Scoring Engine, Risk Parser]
    ↓
[Output: Structured JSON → UI]
```

## 10. אסטרטגיית נתונים ולמידה

### שלב 1 — Bootstrap (MVP)
- **15+ SOWs קיימים** (מהתיקייה הנוכחית) → Prompt examples
- **Scoring Rubric** — מוגדר ידנית על ידי מומחי רכש
- **Risk Taxonomy** — מוגדר ידנית
- **Vendor Seed List** — רשימה ידנית ראשונית לקטגוריית דלק

### שלב 2 — Data Collection (post-MVP)
- כל SOW שנוצר במערכת → בסיס הנתונים
- כל פידבק → labeled dataset
- **פער קריטי שיש לסגור:** נתוני outcome — לאיזה SOWs הגיעו ספקים, אילו עברו אישור, אילו נכשלו. **יש לבנות מנגנון קליטת outcome data מיד עם השקת המוצר.**

### שלב 3 — Continuous Improvement
- Fine-tuning / RAG על בסיס SOWs שסומנו כ-"מצוין"
- A/B testing על שינויי prompt
- Periodic model evaluation

### מדיניות נתונים
- נתוני משתמשים — מוצפנים at rest ובtransit
- SOWs — בבעלות המשתמש; לא משמשים לאימון ללא הסכמה מפורשת
- Audit log לכל פעולה

## 11. אבטחת מידע ורגולציה

### גישה שלב MVP
- פלטפורמה: ענן מסחרי (לא מסווג)
- **SOWs לא יכילו מידע מסווג** — אחריות המשתמש להבין ולאכוף
- חוזה עם ספק ה-AI (Anthropic) יכלול DPA (Data Processing Agreement)
- **Data residency:** מחוץ לישראל בשלב ראשון; לבחון דרישות לפי צורך

### מפת סיכוני אבטחה

| סיכון | פגיעות | מיטיגציה |
|---|---|---|
| דליפת מידע רגיש ל-LLM | משתמש מזין מידע מסווג | הדרכה, disclaimer, ו-data classification detection |
| סריקת אתרים לא חוקית | Web scraper פוגע בתנאי שימוש | כיבוד robots.txt, rate limiting, public sources only |
| גישה לא מורשית | פרצה ב-auth | MFA, JWT rotation, audit logs |
| השפעה על מכרזים | ספק מגלה שה-AI "בנה" את ה-SOW | Privacy by design, anonymization |

### מה יש לבדוק לפני Scale
- עמידה ב-GDPR / ממשל נתונים ישראלי
- בחינת on-premise / air-gapped deployment בשלב מאוחר
- צורת אישור רשמית מגורמי ביטחון מידע

## 12. היקף MVP — קטגוריית דלק

### למה דלק?
- קטגוריה בעלת ייחודיות גבוהה (בטיחות, תנודתיות מחיר, מורכבות לוגיסטית)
- ממחישה את יכולות המערכת טוב יותר מקטגוריה פשוטה
- נתוני שוק דלק נגישים ושפירים

### מה כלול ב-MVP

| מה | פירוט |
|---|---|
| **SOW Generator** | ל-דלק בלבד — תבנית מלאה עם מפרטי בטיחות ואחסון |
| **Scorer** | Rubric מוגדר ידנית, ציון + 3 קטגוריות: שלמות, סיכון, ריאליות |
| **Risk Engine** | Taxonomy ל-דלק: MSDS, אחסון, כוח עליון, תנודתיות מחיר |
| **Market Intelligence** | חיפוש מחירי דלק ישראל + ספקים |
| **Reference Benchmarking** | 2–3 SOWs ממשלתיים לדלק (DoD / NATO) |
| **Feedback** | אגודל למעלה/למטה + הערה |
| **Export** | Word + PDF |

### מה לא כלול ב-MVP
- התממשקות מרק"ט
- Multi-category support
- Fine-tuning / model update pipeline
- Admin dashboard מלא
- Mobile

### הגדרת הצלחה ל-MVP
> משתמש חדש שמעולם לא כתב SOW דלק — מצליח להפיק טיוטה מלאה עם ציון >70 תוך שעה.
